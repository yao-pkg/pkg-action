# Publishing

How `yao-pkg/pkg-action` gets your binaries out of the runner and into
artifacts / releases.

## The two delivery channels

| Channel                                                       | Input                                                        | When to use                                                                                           |
| ------------------------------------------------------------- | ------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------- |
| **Workflow artifact** (`upload-artifact=true`, default)       | `artifact-name` template                                     | Every run. Keeps binaries accessible to downstream jobs and to manual `Actions → Artifacts` download. |
| **GitHub release** (`attach-to-release=true`, off by default) | `release-tag` (auto from `refs/tags/*`) + `release-*` fields | Tagged releases. Attaches the same binaries + SHASUMS sidecars to the release page.                   |

Both channels run off the same finalized file list, so a single workflow
can publish to both simultaneously.

## Minimal release-attach example

```yaml
on:
  push:
    tags: ['v*']

permissions:
  contents: write # required by the release API

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
      - uses: yao-pkg/pkg-action@v1
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        with:
          targets: node22-linux-x64,node22-macos-arm64,node22-win-x64
          compress: tar.gz
          checksum: sha256
          attach-to-release: true # opt in to the release channel
          release-draft: false # publish immediately
          generate-release-table: true # append a size/checksum table
```

The tag comes from `GITHUB_REF` (`refs/tags/v1.2.3`), so no explicit
`release-tag` is needed when the run was triggered by the tag push.

## Explicit release tag (non-tag triggers)

When you want to attach to a release from a `workflow_dispatch` or a
branch push:

```yaml
- uses: yao-pkg/pkg-action@v1
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
  with:
    attach-to-release: true
    release-tag: nightly-${{ github.run_number }}
```

If a release with that tag doesn't exist yet, the action creates one as
a draft and uploads assets into it.

## Release body generation

With `generate-release-table: true` (default), pkg-action appends a
markdown table to the release body — keeping any user-supplied
`release-body` content on top. One row per target:

```
## Your notes here

[...kept as-is...]

## Binaries

| Target | File | Size | Checksum |
| --- | --- | --- | --- |
| `node22-linux-x64` | `tiny-1.0.0-linux-x64.tar.gz` | 8.40 MB | `sha256:c0ffee…f00d` |
| `node22-macos-arm64` | `tiny-1.0.0-macos-arm64.zip` | 8.12 MB | `sha256:cafe…babe` |

<sub>Generated by <code>yao-pkg/pkg-action@1.0.0</code>.</sub>
```

Set `generate-release-table: false` to emit exactly what you pass in
`release-body`.

## Assets uploaded per target

Given `compress: tar.gz` and `checksum: sha256`, the release gets:

- `<name>-<version>-<os>-<arch>.tar.gz` — the archived binary.
- `<name>-<version>-<os>-<arch>.tar.gz.sha256` — the per-file sidecar
  checksum.
- `SHASUMS256.txt` — one consolidated SHASUMS file covering every target.

`replace: true` is always applied: re-running the action against an
existing release overwrites matching assets instead of failing. Upload a
new commit to the same tag and you'll get the new binaries without
manual cleanup.

## Permissions

```yaml
permissions:
  contents: write
```

Required for the release API. Without it, the orchestrator fails at the
attach step with Octokit's 403 message. Signing and metadata-patching
don't need it.

`GITHUB_TOKEN` must be in the `env:` block of the step invoking the
composite — Octokit reads it at that layer. Missing token: the
orchestrator fails before touching Octokit so you never see a confusing 404.

## Interaction with signing

Signing runs BEFORE archive/checksum so the uploaded artifact reflects
the signed bytes, and BEFORE upload so the release asset is the signed
copy:

```
rename → windows-metadata → sign → archive → checksum → upload-artifact → attach-to-release
```

See [docs/signing.md](./signing.md) once that exists — or, for now, the
`signing` category in [docs/inputs.md](./inputs.md).

## Step-summary polish

The job summary (`GITHUB_STEP_SUMMARY`) renders:

- per-target Target / Filename / Size / SHA / Signed / Time
- a **Release:** trailer with the release URL when attach-to-release ran
- action + pkg versions in a meta line

Disable with `step-summary: false`.

## Outputs

In addition to the base outputs (`binaries`, `artifacts`, `checksums`,
`version`), the composite now emits:

| Output        | When set                                                                        |
| ------------- | ------------------------------------------------------------------------------- |
| `release-url` | Only when `attach-to-release=true` succeeded. Absolute URL to the release page. |
