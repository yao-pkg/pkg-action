# Docker / OCI publishing

`yao-pkg/pkg-action` can push a minimal OCI image for every linux
target in the same run that builds the archive. One single-arch image
is pushed per target; when two or more linux archs are built, a
manifest list is created under the canonical tag so `docker pull` picks
up the right arch automatically.

## Opting in

```yaml
on:
  push:
    tags: ['v*']

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
      - uses: yao-pkg/pkg-action@v1
        with:
          targets: node22-linux-x64,node22-linux-arm64
          compress: tar.gz
          docker-image: ghcr.io/${{ github.repository }}:{version}
          docker-username: ${{ github.actor }}
          docker-password: ${{ secrets.GITHUB_TOKEN }}
```

## Required runner tooling

- `docker` CLI with `buildx` — preinstalled on `ubuntu-latest`.
- For multi-arch builds, buildx needs a working QEMU emulator (the
  default runner image ships with one via `tonistiigi/binfmt`). If you
  build only your runner's native arch, no emulator is required.

## Generated Dockerfile

The default minimal layout is:

```Dockerfile
# Generated by yao-pkg/pkg-action — do not edit by hand.
FROM gcr.io/distroless/cc-debian12:latest
COPY my-app /usr/local/bin/my-app
ENTRYPOINT ["/usr/local/bin/my-app"]
```

The `cc` distroless variant bundles `libc`, `libstdc++`, and `libgcc`
— required by the pkg-packaged Node binary. `distroless/base` is
missing `libstdc++.so.6` and will fail with a shared-library load
error at container start.

You can swap the base image with `docker-base-image`, or supply a
fully custom Dockerfile via `docker-dockerfile`. When using a custom
Dockerfile, the build context is the project working directory, so
reference the binary at whatever path you `COPY` it from.

## Image tagging

The `docker-image` input accepts the same tokens as `filename`:

| Token       | Example                |
| ----------- | ---------------------- |
| `{name}`    | `my-app`               |
| `{version}` | `1.2.3`                |
| `{sha}`     | `abc1234`              |
| `{ref}`     | `v1.2.3`               |
| `{tag}`     | `v1.2.3` (tag trigger) |

Per-arch tags are emitted with the `linux/<arch>` suffix collapsed
into a dash — e.g. `ghcr.io/org/app:1.0-amd64`,
`ghcr.io/org/app:1.0-arm64`. The manifest list is then pushed under
the unsuffixed ref.

## Credentials & secret handling

Credentials are written to a scoped `DOCKER_CONFIG` directory under
`RUNNER_TEMP` and handed to `docker buildx` via env. The password
**never** appears on argv — the action emits a bounded
`config.json` file with a base64-encoded `auth` line per registry,
which is the standard Docker credentials format. The orchestrator calls
`core.setSecret` on both `docker-username` and `docker-password` so the
runner masks them in any log output that originates outside the
publisher.

If both inputs are omitted, the publisher assumes the registry accepts
unauthenticated pushes (rare, but useful for local registries and
dry-runs).

## Architecture support

| pkg arch | Docker platform |
| -------- | --------------- |
| `x64`    | `linux/amd64`   |
| `arm64`  | `linux/arm64`   |
| `armv7`  | `linux/arm/v7`  |
| `armv6`  | `linux/arm/v6`  |
| `ppc64`  | `linux/ppc64le` |
| `s390x`  | `linux/s390x`   |

Non-linux targets (macOS, Windows) are silently skipped — a mixed
build simply ends up with archives for those OSes and Docker images
for the linux legs.
